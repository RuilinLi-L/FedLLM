### 核心结论先行
当前工作已经形成 DAGER 攻击、防御 baseline、utility、LRB 机制消融的闭环证据，但强证据边界主要是 SST2+GPT2+batch=2 的 full-gradient DAGER。
Clean FedSGD 泄露严重；LRB、topk、compression 均可在若干点上把当前 DAGER 恢复打到 0，但这不等价于形式化隐私保证。
仅看 full-gradient DAGER，topk@0.1 与 compression@8 仍是更强经验 tradeoff；完整 full_lrb@0.5 已改善 utility，但仍偏重。
最新消融显示 projection bottleneck 是主效应：proj_only@0.5 在单次消融运行中达到 DAGER=0 并保持 clean-level utility，因此下一阶段应把主方法收束为 Projection-LRB。

 
## 一、文档定位与当前工作状态
本文参考 docs/初步思路模版.docx 的研究计划结构，将当前 FedLLM / FL-LLM 方向整理为一份阶段性研究进展文档。它不是重新粘贴已有 Markdown，而是把任务价值、文献与 baseline 位置、LRB 方法机制、已完成实验事实、当前判断和后续计划串成一条可继续写论文的主线。
截至 2026-05-05，项目已经从“复现 DAGER + 枚举 baseline”推进到更清晰的阶段：full-gradient DAGER 风险已被量化，多个 defense 的 privacy/utility 已经有结果，LRB 的主要组件也通过完整消融开始被拆开。当前最重要的变化是，主线不再是继续证明“DAGER 可以被打掉”，而是要证明“哪一种 recoverability bottleneck 能在相近 privacy 下保留更多效用，并且能泛化到 PEFT / partial-gradient 等更广攻击面”。
### 审稿口径下的证据边界
已经完成并可较强支撑的证据：SST2 + GPT2 fine-tuned checkpoint + batch=2 + n_inputs=100 的 full-gradient DAGER attack-time privacy；full training utility fixed points；一次完整 LRB ablation。
尚不能直接外推的证据：LoRA/PEFT 训练期 defense、partial-gradient attack、跨数据集/backbone、多 seed 统计显著性、adaptive attack 与严格 DP accounting。
因此当前论文主张应写成：Projection-LRB 是由 full-gradient DAGER 与消融结果支持的主方法候选，而不是已经在所有 FL-LLM 场景全面优于压缩类 baseline。

### 当前状态一句话
DAGER clean 泄露严重，问题成立；topk/compression 是 full-gradient DAGER 下必须认真比较的强经验 baseline；full_lrb 能强力抑制恢复但过防御；消融显示低分辨率 projection bottleneck 才是当前最值得收束为主方法的机制。

## 二、背景与动机
大模型微调正在从集中式数据汇聚转向联邦学习、跨机构协作训练和 PEFT adapter 共享。这样的设置看起来天然适合隐私场景：医疗文本、金融评论、企业客服、用户输入日志等原始文本不离开本地，只上传梯度、模型更新或轻量参数更新。但梯度反演与文本恢复攻击表明，“不上传原文”并不等价于“原文不可恢复”。在语言模型中，token embedding、layer-wise gradient 和序列损失共同形成了足以暴露样本内容的高分辨率信号。
本项目当前的 clean DAGER 结果给出了直接动机：在 SST2 + GPT2 + batch=2 + n_inputs=100 的设置下，未防御 FedSGD 梯度的 rec_token_mean 达到 0.833506，ROUGE-1+ROUGE-2 达到 141.710856。这说明攻击者并不是只能恢复模糊主题，而是能够恢复大量 token 级文本信息。因此，本工作不是为了在一个 toy setting 中展示攻击存在，而是要面向真实 FL-LLM / PEFT 协作训练中的中间更新泄露问题，寻找可落地的防御机制。
进一步的动机来自 privacy-utility 冲突：简单把 DAGER 打到 0 并不困难，困难在于保留训练效用。当前实验已经显示，完整 LRB 能稳定抑制 DAGER，但 utility 代价偏高；topk/compression 在 full-gradient DAGER 下很强，却更像通信压缩副作用而不是针对文本 recoverability 的机制解释。论文真正需要回答的是：能否设计一种面向 layer-wise recoverability 的 bottleneck，使更新仍可用于学习任务信号，但不再保留可被 DAGER 等攻击恢复的高分辨率文本细节。
## 三、前人工作局限性
已有工作为本项目提供了攻击、隐私和压缩三类重要参照，但各自都留下了适合本工作切入的缺口。当前实验也印证了这些局限：强扰动方法通常牺牲 utility，通信压缩方法在当前 DAGER 下表现强但机制目标不完全对齐，表示扰动和训练侧混合方法在 GPT2/FedSGD 设置中并不稳定。
| 方向 | 代表思路 | 主要局限性 | 对本工作的启发 |
| --- | --- | --- | --- |
| 梯度反演攻击 | DAGER / LAMP 等从梯度恢复文本 | 多数工作强调攻击能力，较少给出适合 FL-LLM 训练流程的低损耗防御；对 PEFT、partial-gradient 和 defense 后 utility 的系统闭环不足。 | 需要把攻击复现、defense、end-to-end utility 和消融放在同一评价链路里。 |
| 噪声与 DP-SGD | 对梯度裁剪后加噪，获得形式化 DP 语义 | 理论上强，但在大模型梯度和小 batch 文本任务中 utility 代价明显；本项目 noise/DP-SGD 结果也显示较难兼顾 privacy 与 accuracy。 | 不能只追求“加噪后看不见”，而要保留任务梯度中的可学习结构。 |
| 通信压缩 | Top-k sparsification、SVD/quantization compression | 当前 full-gradient DAGER 下是强 baseline，但核心目标是少传或压缩，不是解释哪些 layer/direction 泄露文本；跨 PEFT/partial-gradient 攻击面仍未验证。 | 主方法必须正面比较 topk/compression，同时说明 recoverability bottleneck 的机制差异。 |
| 表示扰动与训练增强 | Soteria、mixup 等表示层或训练侧防御 | 迁移到 GPT2/FedSGD 当前设置后不稳定：soteria 几乎完全泄露，mixup 多点 privacy 失败。 | 需要面向 Transformer 梯度结构重新设计，而不是直接搬用 CV/表示防御经验。 |
| PEFT/局部梯度泄露 | LoRA/adapter 更新反演、partial transformer gradient 攻击 | 更贴近真实部署，但当前仓库只有 eval-first 框架和威胁建模，尚缺完整 defense 结果。 | 这是下一阶段证明泛化意义的关键实验面。 |

## 四、希望解决的问题
本工作希望解决的不是单一实验点上的“攻击是否成功”问题，而是 FL-LLM 中间更新如何在可训练与不可恢复之间取得平衡的问题。具体来说，需要把研究问题拆成四个层次：先证明 clean 更新确实泄露，再建立强 baseline 对照，然后设计可解释的 recoverability bottleneck，最后验证该 bottleneck 是否能跨训练形式和攻击面泛化。
| 问题层次 | 要回答的问题 | 当前已有依据 | 下一步需要补强 |
| --- | --- | --- | --- |
| 风险是否成立 | 客户端只上传梯度/更新时，攻击者是否仍能恢复文本？ | clean DAGER rec_token_mean=0.833506，ROUGE-1+ROUGE-2=141.710856。 | 跨数据集/backbone 复核，避免单一 SST2/GPT2 偶然性。 |
| 防御是否有效 | 哪些方法能把 token 恢复和 ROUGE 恢复压到接近 0？ | LRB@0.05~0.5、topk@0.01~0.3、compression@4/8/16 在 DAGER 下表现强。 | 补齐稳定点说明；DAGER=0 只能表示当前攻击未恢复，不等价于严格隐私保证。 |
| 效用是否可接受 | 同等 privacy 下，哪个 defense 保留最多 task accuracy？ | topk@0.1 和 compression@8 是当前 full-gradient DAGER 的强经验 tradeoff；full_lrb utility 偏弱。 | 把 Projection-LRB 作为主方法纳入 main result，与强 baseline 做同等设置和多 seed 比较。 |
| 机制是否清楚 | 防御效果来自 clipping、projection、noise 还是 layer sensitivity？ | ablation 显示 proj_only@0.5 在单次运行中 DAGER=0 且 accuracy=0.915520，是当前最强机制线索。 | 做 projection-only keep-ratio sweep、多 seed 与细消融，确认最宽松 bottleneck 和方差。 |
| 场景是否泛化 | 该机制是否适用于 PEFT/LoRA、partial-gradient 和更真实的联邦微调？ | PEFT eval-first 框架已有，partial-gradient 威胁已建模。 | 补 LoRA 训练期 defense、partial-gradient 攻击入口和跨任务结果。 |

## 五、问题对应的场景与意义
该问题对应的真实场景是多方拥有敏感文本、但又需要共同微调或适配语言模型的协作训练。典型例子包括医院之间联合训练临床文本分类模型、金融机构联合优化舆情或风控文本模型、企业之间共享客服/工单语义能力，以及移动端或边缘端在本地数据上更新个性化语言模型。此时服务端、聚合方或半诚实参与方即使拿不到原文，也可能通过梯度或 adapter 更新进行文本恢复。
| 应用场景 | 为什么会共享更新 | 泄露风险 | 本工作意义 |
| --- | --- | --- | --- |
| 跨机构文本联邦学习 | 机构间数据不能集中，但希望共享模型能力 | 聚合方或参与方可从 batch 梯度推断训练文本 | 为 FL-LLM 提供 privacy audit 与 defense pipeline。 |
| PEFT / LoRA 协作微调 | 只上传 adapter 或低秩参数更新以降低训练成本 | 轻量更新仍可能携带样本级 token 线索 | 验证 defense 是否能贴近未来更常用的轻量微调流程。 |
| 边缘端或个性化模型更新 | 设备端保留用户输入，周期性上传更新 | 短文本、小 batch 和高重复 token 会放大恢复风险 | 提升用户侧文本数据保护能力。 |
| 科研与合规评估 | 需要定量说明某种训练协议是否安全 | 仅报告 accuracy 无法说明更新是否泄露文本 | 建立 privacy-utility-Pareto 和机制消融证据，支撑论文与合规报告。 |

### 本工作的实际意义
从工程上，它给 FedLLM / PEFT 训练流程提供可复现实验链路：攻击复现、defense sweep、utility 训练、消融和后续 PEFT/partial-gradient 扩展。
从论文上，它把问题从“某个 defense 是否让 DAGER 失败”推进到“中间更新的 recoverability 应该如何被结构性压缩，并在相近 privacy 下保留更多任务信号”。
从当前结果看，Projection-LRB / LRB-lite 是最值得继续推进的主方法候选：它直接对应场景中的低损耗隐私更新需求，也与消融证据一致。

## 六、威胁模型与评价对象
本工作关注大模型联邦训练或分布式微调过程中由中间更新信息导致的数据泄露。客户端不上传原始文本，但会上传梯度、模型更新或 PEFT adapter 更新；攻击者可能据此恢复训练样本。这个问题的关键不在于“数据有没有直接共享”，而在于“共享更新本身是否携带足以恢复原始文本的高分辨率信息”。
| 攻击面 | 代表方法 / 文献 | 当前工程状态 | 论文作用 |
| --- | --- | --- | --- |
| Full-gradient inversion | DAGER / LAMP | DAGER 已成为主实验；n_inputs=100 defense sweep 已完成 | 证明 clean FedSGD 梯度存在严重文本恢复风险 |
| PEFT leakage | Gradient Inversion Attacks on PEFT / ReCIT | LoRA eval-first 框架已有，缺完整结果表 | 验证 LRB 是否对实际轻量微调更新也有价值 |

## 七、已有 baseline 与当前创新位置
参考 docs/FL-LLM.md 和 baseline 对比材料，当前 baseline 可以按防御逻辑分成扰动系、隐私标准系、通信压缩系、表示扰动系和训练正则系。它们都能提供必要对照，但大多不是直接以“最小化中间更新的 recoverability”为目标。LRB 的创新位置正在这里：不是粗暴减少数值精度，也不是平均加噪，而是按层和方向建立恢复瓶颈。
| baseline | 主要优势 | 主要不足 | 更适合扮演的角色 |
| --- | --- | --- | --- |
| noise | 实现简单，最小干预 | 需要较大噪声才明显有效，utility 快速下降 | sanity-check baseline |
| DP-SGD | 有明确 DP 语义 | 当前大模型/FedSGD 设置下 utility 代价很大 | 理论标准 baseline |
| topk | 计算/通信成本低，当前 DAGER 下很强 | 不是隐私中心设计，泛化到其他攻击面仍需验证 | 强通信压缩 baseline |
| compression | 兼顾通信效率和信息损失，工程接入容易 | 更多回答少传多少，不直接建模少泄露多少 | 强量化 baseline |
| soteria | 直接触及表示泄露 | 迁移到 GPT2/LLM 当前设置后稳定性差，privacy 反而恶化 | 表示层 baseline / 附录对照 |
| mixup | utility 友好 | 当前 DAGER 下 privacy 明显失败 | 训练侧弱防御 baseline |
| Projection-LRB | 显式针对 layer-wise recoverability bottleneck；消融显示 projection 是主效应 | 多 seed、keep-ratio sweep、PEFT/partial-gradient 证据未补齐 | 当前应收束的主方法候选 |
| full_lrb | 包含 projection、clipping、residual noise，privacy 很强 | 完整配置当前过防御，runtime 偏高 | 强防御 / 过防御消融对照 |

## 八、Projection-LRB 方法机制
根据当前消融证据，文档中的主方法应从泛称 LRB 收束为 Projection-LRB：一个面向 recoverability 的 layer-wise gradient projection defense。full_lrb 仍作为强防御 / 过防御对照保留，用来说明 clipping 与 residual-space noise 并不是当前 full-gradient DAGER 设置下的必要主效应。该实现已经在 train_method=full 的训练 utility 路径中接通，但 LoRA 训练期 defense、forward-side representation bottleneck 和完整 calibration pipeline 仍未落地。
更适合论文的方法表述是：对每层梯度 g_l，先根据结构先验与当前 batch 梯度统计估计敏感度 s_l，再用 keep ratio k_l 控制低分辨率 signed_pool 投影 P_{k_l}(g_l)。Projection-LRB 的核心更新可写作 \tilde{g}_l=P_{k_l}(g_l)，其中高敏感层使用更低分辨率 bottleneck；full_lrb 则在此基础上额外加入 clipping 与 residual-space noise。
| 模块 | 当前实现 | 核心作用 | 消融关注点 |
| --- | --- | --- | --- |
| Layer-wise sensitivity | 结构先验 + 当前 batch 梯度统计；默认 empirical_weight=0.6 | 识别哪些层更可能泄露 token 细节 | rule_only / empirical_only / uniform_all_sensitive |
| Low-resolution signed_pool projection | 先随机符号翻转，再 pooling/interpolation 低分辨率重建 | Projection-LRB 主体；保留粗结构，移除高分辨率样本细节 | proj_only / proj_clip / pool_full |
| Layer-wise clipping | 按所有层范数中位数设置每层裁剪阈值 | full_lrb 中的附加约束；限制个别层的大幅度信息流 | clip_only / proj_clip |
| Residual-space noise | 噪声主要加在低分辨率表示之外的残差方向 | full_lrb 中的强防御组件；进一步污染攻击依赖的细节方向 | full_lrb vs proj_only |

### 推荐当前方法表述
Projection-LRB is a layer-wise adaptive gradient projection defense that suppresses recoverability by mapping sensitive-layer gradients into a low-resolution signed pooling subspace.
full_lrb 可作为包含 clipping 与 residual-space noise 的强防御对照；当前不建议把 full_lrb 写成最终主方法。

| DP 口径 | dpsgd 采用逐样本裁剪 + 高斯噪声的 DP-SGD-style 实现。 | 未做 privacy accountant 时不能声称 epsilon/delta 形式化 DP 保证。 |

| 指标 | 含义 | 解读注意 |
| --- | --- | --- |
| rec_token_mean | DAGER 恢复文本与真实文本在 token 级别的平均恢复程度；越低表示当前攻击恢复出的 token 越少。 | rec_token_mean=0 表示当前攻击和阈值下未恢复 token，不等价于形式化隐私安全。 |

## 十八、可以直接用于汇报的阶段性结论
### 中文结论
当前 SST2+GPT2+batch=2 的 full-gradient DAGER 实验表明，clean FedSGD 梯度存在严重文本恢复风险。full_lrb 完整配置能稳定把 DAGER 恢复压到 0，但其 clipping 和 residual noise 带来了明显 utility 代价。消融进一步显示，低分辨率 signed_pool projection 是当前 LRB 有效性的主因：proj_only@0.5 在不加噪、不做完整裁剪的情况下即可实现 DAGER=0，并在单次运行中保持 clean-level accuracy。因此，下一阶段应把主方法身份统一为 Projection-LRB / LRB-lite，并通过 keep-ratio sweep、多 seed、LoRA/PEFT、partial-gradient 和跨数据集实验验证其泛化价值。

### English statement
In the current SST2/GPT2 full-gradient DAGER setting, the dominant effective component is the low-resolution signed projection bottleneck. While the full LRB configuration suppresses token recovery to zero, its clipping and residual-space noise introduce unnecessary utility loss. The projection-only variant achieves zero token and ROUGE recovery with clean-level utility in the current ablation run, making Projection-LRB the most promising method candidate pending multi-seed and cross-attack validation.

## 一句话判断：题目有价值，方法有苗头，但论文必须“换主线”

从你给出的材料看，**真正有投稿价值的，不是“LRB 打败了 DAGER”**，而是下面这条更强、也更诚实的主线：

> **在 FL-LLM / PEFT 场景中，文本泄露的关键不是更新值本身有多大，而是更新里是否保留了可恢复的高分辨率子空间；Projection-LRB 通过 layer-wise low-resolution signed projection，主动切断 recoverability bottleneck，在保持 clean-level 或 near-clean utility 的同时抑制多类文本恢复攻击。**

这条线比“加一点防御以后 DAGER=0”强很多，因为 DAGER 本身已经把问题门槛拉高了：它不是模糊恢复，而是利用 Transformer attention gradient 的低秩结构做 exact recovery，并且在 GPT-2 / BERT / LLaMA-2、多数据集、较大 batch 上都展示了强恢复能力；论文还明确指出它适用于 FedSGD、LoRA finetuning 等设置。也就是说，审稿人不会怀疑“攻击是否成立”，他们会怀疑的是：**你的防御有没有提出真正新的、机制上可解释的瓶颈，而不只是碰巧让某个 attack fail。** [DAGER](https://proceedings.neurips.cc/paper_files/paper/2024/file/9ff1577a1f8308df1ccea6b4f64a103f-Paper-Conference.pdf)

## 从严格审稿人的角度，你们目前最强的地方在哪里

我先说优点，而且是真优点。

第一，**问题成立得非常扎实**。你们不是在做“可能有一点泄露”的泛泛讨论，而是已经在一个明确设置上把 clean 泄露量化出来了：SST2 + GPT2 + batch=2 + full-gradient DAGER 下，clean FedSGD 有非常高的 token-level 与 ROUGE 恢复。这和现有攻击文献的趋势完全一致：DAGER 已经证明 transformer/LLM 梯度在 honest-but-curious 条件下可以 exact recover；partial-gradient 工作进一步说明，甚至单层、单模块、极少参数暴露都可能泄露文本；而最新的 PEFT 攻击又把风险推进到 LoRA / adapter 这样的真实轻量微调场景。也就是说，你们选的题目并不是“攻击已过时、没人关心”的方向，反而是一个在 2024–2026 明显升温的前沿问题。 [DAGER](https://proceedings.neurips.cc/paper_files/paper/2024/file/9ff1577a1f8308df1ccea6b4f64a103f-Paper-Conference.pdf) [Partial Transformer Gradients](https://aclanthology.org/2024.emnlp-main.275.pdf) [ReCIT](https://arxiv.org/pdf/2504.20570)

第二，**你们已经有了一个像“方法论文”而不是“工程报告”的机制线索**。最重要的数据不是 full_lrb 全家桶，而是你们自己的 ablation：`proj_only@0.5` 在单次运行里已经做到 DAGER=0 且 utility 接近 clean，反而 `clip_only` 基本无效、`full_lrb` 还过防御。这种结果非常关键，因为它说明你们不是“各种招都加一点然后碰巧有效”，而是**projection bottleneck 才是主效应**。如果把这个讲顺，论文会从“防御组合技”升级为“recoverability bottleneck 的机制发现 + defense design”。

第三，**你们 baseline 意识是对的**。很多这类论文的死法，是只跟 noise / DP-SGD / Soteria 这种“好打的 baseline”比，而回避 compression / top-k 这种现实里更强的经验对手。你们已经意识到 topk@0.1 和 compression@8 才是 full-gradient DAGER 下真正难打的 baseline，这会让文章更可信。AAAI 审稿也明确看 baseline 是否合适、比较是否充分。 [AAAI Reviewer Instructions](https://aaai.org/conference/aaai/aaai-26/instructions-for-aaai-26-reviewers/)

## 但从 AAAI 的标准看，当前最危险的短板是什么

### 1. 最大风险：新意还没有被“锚定”为一个不可替代的 claim

如果你们继续把论文写成“LRB defense against DAGER”，那我觉得很危险。因为严格审稿人会立刻问三句：

1. **为什么不是 top-k / quantization 的 another variant？**  
2. **为什么不是一个调得更合适的压缩 baseline 就能做到？**  
3. **为什么只在 SST2/GPT2/full-gradient 上成立？**

这三句一旦成立，paper 就会被打成“经验性 defense tweak”。

所以你们必须把主张收缩成一个更清楚的**最小可 defend claim**：

> **不是 full_lrb，而是 Projection-LRB。**  
> 它的核心不是加噪或裁剪，而是对 sensitive-layer gradients 施加低分辨率 signed projection，从而显式切断文本可恢复的高频细节；这种 bottleneck 比通用压缩更贴近泄露机制，因此有机会在跨攻击面时更稳定。

如果你们不把主方法从 “LRB” 改成 “Projection-LRB / LRB-lite”，我觉得会持续被自己的 ablation 反噬。

### 2. 第二大风险：证据边界太窄，容易被批“single-setting story”

你自己已经写得很诚实了：目前强证据主要是 **SST2 + GPT2 + batch=2 + full-gradient DAGER**。这在内部推进足够，但在 AAAI 主会里不够撑大 claim。

为什么？因为相关工作已经把攻击面扩展得很厉害了。DAGER 论文自己就覆盖了 encoder / decoder、多个数据集、较大 batch，并提到可用于 LoRA；partial-gradient 论文则直接证明，仅暴露单层甚至单模块梯度也能有效恢复；ReCIT 又把 PEFT 场景里的 prefix + PII 恢复推到了更接近真实部署的威胁模型。这样一来，审稿人会天然要求你们至少跨 **训练形式** 或 **攻击面** 做一点验证，否则会觉得你们的 defense 只是在一个固定 benchmark 上打 attack-specific loophole。 [DAGER](https://proceedings.neurips.cc/paper_files/paper/2024/file/9ff1577a1f8308df1ccea6b4f64a103f-Paper-Conference.pdf) [Partial Transformer Gradients](https://aclanthology.org/2024.emnlp-main.275.pdf) [ReCIT](https://arxiv.org/pdf/2504.20570)

### 3. 第三大风险：目前最强 baseline 仍然不是你们

这是个必须直说的地方：**按你目前给的 full-gradient DAGER 结果，topk@0.1 与 compression@8 仍然是最强经验 tradeoff。**  
这不是坏消息，但它决定了论文该怎么写。

如果论文标题和主张是 “our method achieves best privacy-utility tradeoff”，我会建议别这么写，容易被一票否掉。  
如果主张改成：

> **Projection-LRB offers a mechanism-grounded recoverability bottleneck, and shows its value especially when moving beyond a single full-gradient attack setting.**

那就合理很多。因为这时你不是在和 top-k 比“谁更像一个压缩器”，而是在比**谁更像一个可解释、可迁移的隐私机制**。

### 4. 第四大风险：缺 adaptive/transfer attack，DAGER=0 不足以说服审稿人

严格审稿人一定会说：  
“你现在只能证明这个 defense 让**当前这版 DAGER**失败，不代表真的 privacy stronger。”

这个问题不能靠文字解释过去，至少要补一个实验层面的回应。最低要求是：

- 不止跑 DAGER；
- 再跑一个 optimization-based 文本恢复攻击，哪怕是 LAMP/TAG 风格的较弱攻击；
- 或者做一个“defense-aware / attack-aware”版本的恢复尝试。

partial-gradient 论文里已经展示了 optimization-based 攻击在不同梯度粒度下仍然有效；而 ReCIT 也说明在 PEFT 里单靠 analytic prefix recovery 不是全部，攻击者可以混合其它策略。换句话说，**单一攻击失败 ≠ 机制安全**，这是你们必须主动承认并实验回应的。 [Partial Transformer Gradients](https://aclanthology.org/2024.emnlp-main.275.pdf) [ReCIT](https://arxiv.org/pdf/2504.20570)

### 英文摘要草案

**Abstract**  
Federated fine-tuning of large language models (LLMs) is increasingly adopted in privacy-sensitive domains, yet recent gradient inversion attacks show that shared updates can reveal substantial token-level training content. Existing defenses mainly rely on generic perturbation, clipping, or communication-oriented compression, which either incur large utility loss or provide limited insight into why textual recoverability is suppressed. In this paper, we argue that the key object to defend is not gradient magnitude alone, but the high-resolution recoverable subspace encoded in layer-wise updates. We introduce **Projection-LRB**, a layer-wise adaptive defense that maps sensitive gradients into a low-resolution signed pooling subspace, thereby imposing a **recoverability bottleneck** while preserving coarse task-relevant learning signals. Compared with strong empirical baselines such as top-k sparsification and gradient compression, Projection-LRB offers a more mechanism-grounded privacy-utility tradeoff and forms a stronger candidate for transfer to PEFT and partial-gradient leakage settings. These results position Projection-LRB as a promising defense direction for privacy-preserving federated LLM fine-tuning beyond attack-specific patching. ### FedLLM / Projection-LRB：AAAI 论文一页式卖点总结

#### 1. 我们要解决什么问题
联邦 LLM 微调与 PEFT 看似避免了原始文本共享，但共享的梯度/更新本身可能泄露 token 级训练内容。近期强攻击已经从 full-gradient 扩展到 partial-gradient 与 PEFT，说明“仅传更新”并不足以提供文本隐私。 [DAGER](https://proceedings.neurips.cc/paper_files/paper/2024/file/9ff1577a1f8308df1ccea6b4f64a103f-Paper-Conference.pdf) [Partial Transformer Gradients](https://aclanthology.org/2024.emnlp-main.275.pdf) [ReCIT](https://arxiv.org/pdf/2504.20570)

#### 2. 现有方法为什么不够
现有防御要么依赖通用噪声/DP，往往 utility 代价高；要么依赖 top-k / quantization 之类的压缩手段，虽然在某些设置下经验上很强，但其核心目标是通信效率，不是解释和切断文本 recoverability 的来源。换句话说，现有方法大多在“处理梯度”，但没有直接处理“可恢复结构”。

#### 3. 我们的核心 insight
文本泄露的关键不是更新值有多大，而是更新里是否保留了**支持 token 恢复与序列重建的高分辨率子空间**。因此，防御的目标应从 generic perturbation 转向 **recoverability bottleneck design**。

#### 4. 我们的方法是什么
**Projection-LRB** 对每层梯度施加低分辨率 signed projection：  
对敏感层，用更强的低分辨率瓶颈压缩高频可恢复信息；  
对非敏感层，尽量保留粗粒度任务相关结构。  
其目标不是“让梯度变小”，而是“让梯度不再适合文本恢复”。

#### 5. 我们目前最强的机制证据是什么
在当前 full-gradient DAGER 设置下，系统消融显示：
- `clip_only` 几乎不能阻断恢复；
- `full_lrb` 虽然能压低恢复，但 utility 偏重；
- `proj_only` 已能在单次运行中实现 DAGER=0 且保持 clean-level utility。  
这说明 **projection bottleneck 才是主效应**，方法主角应从 full LRB 收束为 **Projection-LRB**。 [DAGER](https://proceedings.neurips.cc/paper_files/paper/2024/file/9ff1577a1f8308df1ccea6b4f64a103f-Paper-Conference.pdf)

#### 6. 这篇论文真正的新意是什么
这篇论文的新意不该表述为“又一个隐私 defense”，而应表述为：
- 提出 **recoverability bottleneck** 这一更贴近 LLM 泄露机制的问题刻画；
- 证明 **low-resolution signed projection** 是当前最主要的有效机制；
- 将 defense 从 attack-specific patching 提升为可解释的结构性压缩设计。

#### 7. 为什么 AAAI 会关心
AAAI 审稿强调：问题是否重要、技术贡献是否清晰、实验是否真正支持 claims、baseline 是否充分、结论边界是否诚实。你们这个故事在问题重要性上天然强，只要把方法主张收束为 Projection-LRB，并补齐 calibration、强 baseline 比较和跨攻击面验证，就符合 AAAI 喜欢的“问题明确 + 机制清楚 + 实证扎实”的论文形态。 [AAAI Reviewer Instructions](https://aaai.org/conference/aaai/aaai-26/instructions-for-aaai-26-reviewers/) [AAAI Review Process](https://aaai.org/conference/aaai/aaai-26/review-process/)

#### 8. 论文应如何诚实定位

Second, we propose **Projection-LRB**, a layer-wise adaptive defense based on low-resolution signed projection, designed to preserve coarse task-relevant learning signals while removing high-resolution information exploitable by text reconstruction attacks. Third, through systematic ablations, we show that the projection component is the dominant effective factor in the current full-gradient setting, whereas clipping and residual-space noise are not the primary drivers of privacy gains. Finally, we position Projection-LRB against strong empirical baselines under a DAGER-based threat model and argue that its mechanism-grounded design makes it a stronger candidate for transfer to PEFT and partial-gradient leakage settings than generic compression alone. ### 9. Discussion, Limitations, and Broader Impact

#### 这一节要写什么
这节很重要，尤其对 AAAI。  
AAAI reviewer instructions 明确要求论文说明局限、generalizability 和实验是否真的支撑 claims。[AAAI Reviewer Instructions](https://aaai.org/conference/aaai/aaai-26/instructions-for-aaai-26-reviewers/)

建议写成三个短小节：

#### 9.1 What We Claim
明确说：
- 我们验证了 Projection-LRB 在当前 threat model 下的有效性
- 我们证明了 projection 是 dominant component
- 我们不声称 formal privacy guarantee

#### 9.2 What We Do Not Claim
明确写：
- not equivalent to differential privacy
- not proven secure under adaptive attacks
- not yet universally superior to top-k/compression in every regime
- transfer still depends on future broader experiments

#### 9.3 Broader Impact
写两面性：
- 正面：有助于联邦 LLM / PEFT 的隐私保护
- 风险：攻击研究本身也可能被滥用，因此我们强调责任披露和防御导向

#### 这一节的核心论点
**核心论点：我们提供的是机制性防御方向而非形式化隐私证明，并且诚实界定了当前证据边界。**

#### 这一节建议放的表或图
一般不需要。  
如果你们想加一个总结框，可以做：

### 10. Conclusion

#### 这一节要写什么
结论不要再总结所有实验细节，而是回到论文层面的三个 takeaways：

1. clean federated LLM updates leak severe textual information  
2. projection bottleneck is the dominant effective factor  
3. Projection-LRB is a promising mechanism-grounded defense direction for federated LLM fine-tuning beyond attack-specific patching

#### 这一节的核心论点
**核心论点：Projection-LRB 重新定义了你们论文的贡献——不是 another defense trick，而是 recoverability bottleneck 的结构性设计与验证。**

通常不放。

## 五、哪些内容放主文，哪些放附录

这个很关键。AAAI 主文空间紧，不能什么都塞。

### 主文必须保留
- clean leakage anchor
- full-gradient main results
- Projection-LRB method definition
- ablation proving projection is dominant
- 至少一个 transfer 结果（PEFT 或 partial-gradient）
- limitations / non-claims

### 附录适合放
- 超参数细节
- 更多 seeds
- 更多 dataset/backbone
- 完整 baseline sweep
- 更多恢复样例
- 更多 runtime logs
- 实现细节、伪代码扩展
- full_lrb / rule-only / empirical-only 的更细 variant

附录的作用是**增强可信度**，不是承载主论点。主论点必须在主文里闭环。

## 七、每一节最需要避免的坑

### Introduction 的坑
不要写成“我们全面超过所有 baseline”。  
你们目前最强的表述应该是：**Projection-LRB 是机制更清晰、由消融支持的主方法候选**。

### Threat Model 的坑
不要把未验证场景写得像已验证结论。  
尤其 PEFT / partial-gradient，如果结果还不完整，语气要克制。

### Method 的坑
不要让 full_lrb 继续喧宾夺主。  
要明确：full_lrb 是 broader family / strong-defense reference，Projection-LRB 才是 main method。

### Main Results 的坑
不要只报 DAGER=0。  
必须同时报 utility 和 runtime。

### Ablation 的坑
不要只说 “proj_only 最好”。  
要解释为什么 clip_only、full_lrb、uniform 等结果支持你们的机制结论。

### Transfer 的坑
不要把 transfer 写成“大而全的 benchmark”。  
重点是证明：recoverability bottleneck 不是单一 full-gradient artifact。

### Discussion 的坑
不要回避 formal privacy guarantee 缺失。  
相反，正面承认，会更可信。

## 八、我建议的“主文写作优先级”

如果你们现在就要开始写，我建议按下面顺序推进：

第一优先，先写 **Introduction + Threat Model + Method**。  
因为这三节会决定全篇是不是在讲同一个故事。

第二优先，写 **Main Results + Ablation**。  
这是论文的证据中心。

第三优先，写 **Transfer + Discussion**。  
这两节负责让 paper 从“可投”变成“更有说服力”。

最后写 **Abstract + Conclusion**。  
等你们把主图主表都定了再写会更稳。
