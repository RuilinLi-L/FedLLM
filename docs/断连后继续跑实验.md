# 断连后继续跑实验

这篇说明文档解决的是一个很具体的问题：你的本地电脑断电、断网，或者 SSH 连接断开之后，服务器上的 utility / 训练实验怎样继续跑，不因为终端断开而中止。

## 1. 先分清两种“断掉”

先区分两种情况。

情况 A：本地电脑断电、断网，或者 SSH 会话断开。

情况 B：远端服务器自己断电、重启，或者被管理员清理进程。

`tmux` 和 `nohup` 主要解决的是情况 A，也就是“你和服务器的连接断了，但服务器本身还活着”。如果是情况 B，进程通常还是会中断。这时需要更稳定的服务器、任务调度系统，或者集群上的 `sbatch` / `slurm`。

## 2. 最推荐方案：用 `tmux`

如果服务器装了 `tmux`，这是最推荐的做法。它最适合 SSH 跑长实验的场景：你可以先启动实验，再从会话里脱离；本地电脑关机或网络断开后，实验还会继续在服务器上跑；重新连回服务器后，还可以接回原来的终端界面。

### 2.1 登录服务器并进入项目

```bash
ssh <user>@<server>
cd <你的仓库目录>
conda activate dager
```

### 2.2 新建一个 `tmux` 会话

```bash
tmux new -s utility
```

这里的 `utility` 是会话名，你也可以换成别的名字，比如 `sst2_run` 或 `train_gpt2`。

### 2.3 在 `tmux` 里启动实验

下面是一条和当前仓库匹配的完整 utility 示例命令：

```bash
bash scripts/utility_baselines.sh sst2 2 gpt2 1 \
  --anchor_dir ./models/gpt2-ft-rt \
  --privacy_logs log/runs/test
```

如果你现在只想先跑一个 baseline，减少总耗时，可以用：

```bash
bash scripts/utility_baselines.sh sst2 2 gpt2 1 \
  --anchor_dir ./models/gpt2-ft-rt \
  --privacy_logs log/runs/test \
  --baseline_defense dpsgd
```

### 2.4 从会话里脱离，但不停止实验

按下面的按键顺序：

```text
Ctrl+b，然后按 d
```

执行后你会回到普通 shell，但 `tmux` 会话里的实验会继续跑。这时即使你关掉电脑、合上笔记本，或者 SSH 连接断开，只要服务器本身没问题，这个实验还是会继续执行。

### 2.5 重新连接后恢复会话

下次重新连接服务器后：

```bash
ssh <user>@<server>
tmux ls
tmux attach -t utility
```

如果你之前的会话名不是 `utility`，把最后一条命令里的名字改掉即可。

## 3. 备选方案：用 `nohup ... &`

如果服务器没有 `tmux`，或者你只想“挂后台跑着”，不关心之后是否能回到同一个交互界面，可以用 `nohup`。

```bash
cd <你的仓库目录>
conda activate dager
nohup bash scripts/utility_baselines.sh sst2 2 gpt2 1 \
  --anchor_dir ./models/gpt2-ft-rt \
  --privacy_logs log/runs/test \
  > log/runs/manual_utility_$(date +%Y%m%d_%H%M%S).txt 2>&1 < /dev/null &
```

这个方案更像“纯后台执行”。命令启动后会立刻回到 shell，断开 SSH 后进程通常仍会继续，但它不像 `tmux` 那样能让你重新接回同一个终端界面。所以如果你的目标只是“别因为断连停掉实验”，`nohup` 可以用；如果你还想方便地中途查看输出、继续交互，优先用 `tmux`。

## 4. 怎么看它还在不在跑，以及日志写到了哪

先看 `tmux` 会话还在不在：

```bash
tmux ls
```

如果能看到你之前创建的 `utility` 会话，说明会话还在。

再看进程还在不在：

```bash
ps -ef | grep utility_baselines
```

如果能看到对应的 `bash scripts/utility_baselines.sh ...` 或相关 `python3 train.py` 进程，说明实验还在跑。

当前仓库已经自带自动日志能力。运行 `scripts/` 下面的辅助脚本时，终端输出会自动写入 `log/runs/`。此外，`scripts/utility_baselines.sh` 还会创建一个单独的 run 目录，例如：

```text
log/runs/utility_baselines_sst2_b2_gpt2_YYYYMMDD_HHMMSS/
```

可以先看有哪些 run：

```bash
ls log/runs
```

然后用 `tail -f` 持续看输出：

```bash
tail -f log/runs/manual_utility_*.txt
```

或者直接看某个 utility run 里的子日志，例如：

```bash
tail -f log/runs/utility_baselines_*/train_none_seed101.txt
```

也可以看退出码汇总：

```bash
tail -f log/runs/utility_baselines_*/exit_codes.csv
```

如果日志还在持续增长，或者 `exit_codes.csv` 还没有全部结束，就说明任务仍在运行。

## 5. 仓库内推荐直接复制的命令模板

推荐的完整 utility 命令如下：

```bash
bash scripts/utility_baselines.sh sst2 2 gpt2 1 \
  --anchor_dir ./models/gpt2-ft-rt \
  --privacy_logs log/runs/test
```

只跑一个 baseline 的缩短版如下：

```bash
bash scripts/utility_baselines.sh sst2 2 gpt2 1 \
  --anchor_dir ./models/gpt2-ft-rt \
  --privacy_logs log/runs/test \
  --baseline_defense dpsgd
```

把它放进 `tmux` 里的最常见组合如下：

```bash
ssh <user>@<server>
cd <你的仓库目录>
conda activate dager
tmux new -s utility

bash scripts/utility_baselines.sh sst2 2 gpt2 1 \
  --anchor_dir ./models/gpt2-ft-rt \
  --privacy_logs log/runs/test
```

跑起来后按：

```text
Ctrl+b，然后按 d
```

## 6. 常见误区

### 误区 1：我把本地笔记本合上，服务器上的任务一定会停

不一定。如果任务是在 `tmux` 里启动的，或者用 `nohup ... &` 挂到了后台，那么本地电脑关机通常不会直接杀掉服务器上的进程。真正重要的是服务器还在不在，以及你的任务是不是已经和当前 SSH 前台会话解耦。

### 误区 2：我直接在普通 SSH 终端里前台运行，也能一样稳定

不行。如果你只是普通地执行下面这条命令，然后 SSH 会话断开，前台进程往往也会跟着结束：

```bash
bash scripts/utility_baselines.sh ...
```

要避免这个问题，应该提前用 `tmux` 或 `nohup`。

### 误区 3：`tmux` / `nohup` 能防服务器断电

不能。这两种方法只能防“本地断连”，不能防“远端服务器自己掉电、重启、被清理进程”。如果你的环境是实验室集群或学校公共平台，长期任务更推荐使用调度系统，比如 `sbatch`。

### 误区 4：只要命令还在，结果就一定会自动整理好

不一定。虽然 `scripts/utility_baselines.sh` 会在结束时自动汇总结果，但如果任务中途异常退出，最终的 `results.csv`、`utility_results.csv`、`privacy_utility_tradeoff.csv` 可能并不完整。所以长任务最好定期看一下：

```bash
ls log/runs
tail -n 50 log/runs/utility_baselines_*/exit_codes.csv
```

## 7. 一句话建议

如果你现在的目标只是“宿舍断电断网后，服务器继续跑 utility 实验”，最省心的做法就是：SSH 登录服务器，执行 `conda activate dager`，然后 `tmux new -s utility`，在里面运行 `bash scripts/utility_baselines.sh ...`，最后按 `Ctrl+b` 再按 `d`。

如果你的服务器是集群环境，不是普通独占机器，那么下一步应该考虑把实验改成 `sbatch` 提交，而不是长期依赖交互式 SSH 会话。
