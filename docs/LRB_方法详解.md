# LRB 方法详解

这份文档解释的是当前仓库里已经实现并接入的 `LRB` 方法，也就是 `Layer-wise Recoverability Bottleneck`。它的核心目标不是“专门骗过某一种攻击”，而是更一般地让客户端上传的梯度更新变得**更难被反推出原始样本**，同时尽量保留训练所需的有效信息。

为了避免概念混淆，先说清楚一件事：

- 这份文档的主解释对象，是仓库当前代码里的 `LRB` 实现。
- 文档中也会顺带解释 `FL-LLM.md` 和 `DAGER_general_defense_notes.md` 里更理想化、更研究化的 `LRB-v2 / HLRB` 叙事。
- 二者方向一致，但**不完全等价**。当前代码实现的是一个已经可运行的、以梯度后处理为核心的版本；它不是完整训练时版本，也不是严格意义上的理论最优解。

## 1. 一句话先说明白

`LRB` 可以概括成下面这句话：

> 对每一层梯度，先判断它有多“危险”，再对危险层做更强的裁剪、更强的低分辨率投影，并且主要在被丢弃的细节方向上加噪，从而让上传更新只保留“够训练用”的粗结构，而尽量不保留“足够还原样本”的高分辨率细节。

如果用更通俗的话说：

> 它不是把所有梯度一股脑打乱，而是有选择地把“最容易泄露隐私的层”和“最容易泄露细节的方向”处理得更重一些。

## 2. 它在这个仓库里到底是什么

当前仓库里的 `LRB` 主要是一个**梯度后处理防御**：

- 核心实现文件是 `utils/lrb_defense.py`。
- 统一防御入口在 `utils/defenses.py`，当 `defense == "lrb"` 时会调用 `apply_lrb_defense(...)`。
- 参数定义在 `args_factory.py`。
- baseline 语义说明在 `DEFENSE_BASELINES.md`。
- `LRB_EXPERIMENT_RUNBOOK.md` 明确写了：当前 `train.py` 还没有 `--defense lrb` 的训练时 hook。

所以当前版本的准确定位是：

- 它已经能在现有攻击/评估流程里对梯度做防御变换。
- 但它还不是一个已经完整接入训练主循环的“训练时 LRB”版本。

这点很重要。严谨地说，当前仓库里的 `LRB` 是：

> 一个基于层级敏感度、自适应压缩和残差方向加噪的 post-gradient transform。

## 3. LRB 想解决的根本问题是什么

先抽象一下。

设客户端本地原始数据为 \(X_c\)，原始梯度更新为

\[
G_c = (G_{c,1}, G_{c,2}, \dots, G_{c,L}),
\]

其中 \(G_{c,l}\) 表示第 \(l\) 层参数对应的梯度张量。

LRB 的作用是构造一个防御变换

\[
\widetilde G_c = F_\phi(G_c),
\]

使得：

- \(\widetilde G_c\) 仍然保留足够多的任务相关信息；
- 但 \(\widetilde G_c\) 相比原始 \(G_c\) 更难被攻击者拿来恢复输入样本 \(X_c\)。

如果用一个抽象目标来写，可以理解成：

\[
\min_\phi \ \mathcal L_{\text{utility}}(\widetilde G_c) + \lambda \,\mathcal R_{\text{recoverability}}(\widetilde G_c).
\]

这里：

- \(\mathcal L_{\text{utility}}\) 表示任务效用损失；
- \(\mathcal R_{\text{recoverability}}\) 表示“更新可恢复性”的风险；
- \(\lambda\) 是隐私和效用之间的权衡系数。

不过要特别注意：

- 这是**概念上的目标函数**；
- 当前代码并不是显式求解这个优化问题；
- 它实现的是一个工程上可运行的启发式近似方案。

所以，严格来说，当前实现不是“从一个封闭形式目标推导出来的最优算法”，而是一个**围绕 recoverability bottleneck 思想设计出来的结构化启发式防御**。

## 4. 当前实现的完整机制

当前 `LRB` 的主流程可以分成五步：

1. 为每一层估计一个敏感度分数。
2. 根据敏感度决定该层保留多少分辨率、裁剪多狠、加多少噪声。
3. 先做按层裁剪。
4. 再把梯度压到一个低分辨率的公共表示里。
5. 最后主要向“被丢掉的细节方向”加噪。

下面逐步展开。

## 5. 第一步：按层估计敏感度

### 5.1 为什么要分层

并不是所有层的梯度都一样危险。

在大模型和 transformer 场景里，通常更容易泄露输入细节的层包括：

- embedding 层；
- 前几层 transformer block；
- 尤其是早层 attention 相关参数。

而一些输出头、bias、LayerNorm 之类的参数，通常没有那么高的“样本可恢复性”。

因此 LRB 不采用“一刀切”的处理方式，而是先给每层打一个“隐私敏感度分数”。

### 5.2 当前代码里的结构先验

在 `utils/lrb_defense.py` 里，`_layer_sensitivity(...)` 先根据层名给出一个**启发式先验分数** \(p_l\)。

大致规则如下：

- embedding 类参数：\(p_l = 1.0\)
- 前 `sensitive_n_layers` 层里的 attention 类参数：\(p_l = 1.0\)
- 前 `sensitive_n_layers` 层里的其他参数：\(p_l = 0.7\)
- 其他 attention 类参数：\(p_l = 0.45\)
- 分类头、`lm_head`、bias、LayerNorm 一类参数：\(p_l = 0.15\)
- 其他普通参数：\(p_l = 0.25\)

这里的设计直觉是：

- embedding 和早层 attention 更直接承载 token 级别的信息；
- 它们更容易与原始输入样本形成强对应关系；
- 所以默认应该保护得更重。

### 5.3 当前代码里的经验校准

仅靠层名规则还不够，所以当前实现又加了一层**轻量经验校准**。

对每个有效梯度张量 \(G_l\)，代码会计算三类统计量：

1. 梯度范数大小

\[
m^{(\text{norm})}_l = \log(1 + \|G_l\|_2).
\]

2. 低分辨率投影后的残差比例

先从梯度里抽样一个子向量或子张量 \(S_l\)，再做一次低分辨率近似 \(T(S_l)\)，然后计算：

\[
m^{(\text{res})}_l = \frac{\|S_l - T(S_l)\|_2}{\|S_l\|_2 + \varepsilon}.
\]

这个量越大，说明该层梯度里有更多内容**无法被低分辨率结构表示**，也就意味着里面可能含有更多细粒度、样本特异的信息。

3. 尖峰性（spikiness）

\[
m^{(\text{spike})}_l
=
\log\!\left(1 + \frac{\max_i |(G_l)_i|}{\operatorname{RMS}(G_l) + \varepsilon}\right).
\]

它衡量的是梯度是否特别“尖”、特别集中。

然后，这三类统计量会分别在当前各层之间做 min-max 归一化，得到

\[
\hat m^{(\text{norm})}_l,\quad
\hat m^{(\text{res})}_l,\quad
\hat m^{(\text{spike})}_l.
\]

再按固定权重合成经验分数：

\[
e_l
=
0.45\,\hat m^{(\text{norm})}_l
+ 0.40\,\hat m^{(\text{res})}_l
+ 0.15\,\hat m^{(\text{spike})}_l.
\]

### 5.4 最终敏感度分数

最后，把结构先验和经验校准按权重混合：

\[
s_l = (1-w)\,p_l + w\,e_l,
\]

其中：

- \(s_l \in [0,1]\) 是最终敏感度；
- \(w = \texttt{defense\_lrb\_empirical\_weight}\)；
- 默认 \(w = 0.6\)。

这意味着：

- 当 \(w=0\) 时，只用层名启发式；
- 当 \(w=1\) 时，只用当前梯度统计；
- 默认值 `0.6` 表示更偏向经验统计，但仍保留结构先验。

### 5.5 严谨理解这一步

这一层混合的本质不是“准确预测哪层一定泄露”，而是做一个**风险排序**：

- 哪些层更值得重保护；
- 哪些层可以少伤一点效用；
- 哪些层应该保留更高分辨率。

从算法设计上看，这是一种**layer-wise adaptive defense schedule**。

## 6. 第二步：由敏感度决定三种处理强度

得到每层敏感度 \(s_l\) 之后，当前实现不会只改一个参数，而是同时决定三类量：

- 保留比例 `keep_ratio`
- 裁剪强度 `clip_scale`
- 噪声强度 `noise_scale`

它们都通过同一个线性插值公式得到。

设：

- 敏感层配置为 \(a_{\text{sens}}\)
- 非敏感层配置为 \(a_{\text{other}}\)

则第 \(l\) 层的实际值为：

\[
a_l = (1-s_l)\,a_{\text{other}} + s_l\,a_{\text{sens}}.
\]

于是就有：

\[
r_l = (1-s_l)\,r_{\text{other}} + s_l\,r_{\text{sens}},
\]

\[
c_l = (1-s_l)\,c_{\text{other}} + s_l\,c_{\text{sens}},
\]

\[
\sigma_l = (1-s_l)\,\sigma_{\text{other}} + s_l\,\sigma_{\text{sens}}.
\]

其中：

- \(r_l\) 是该层的保留比例；
- \(c_l\) 是裁剪缩放因子；
- \(\sigma_l\) 是噪声倍率。

默认参数下，敏感层大致会表现为：

- 更低的 `keep_ratio`
- 更小的裁剪阈值
- 更大的噪声倍率

这正对应“越危险，处理越重”。

## 7. 第三步：按层裁剪

当前实现先求所有层梯度范数的中位数：

\[
M = \operatorname{median}_l \|G_l\|_2.
\]

然后为每一层设置裁剪阈值：

\[
C_l = M \cdot c_l.
\]

接着做裁剪：

\[
\widehat G_l
=
\operatorname{clip}(G_l, C_l)
=
\begin{cases}
G_l, & \|G_l\|_2 \le C_l, \\
\dfrac{C_l}{\|G_l\|_2+\varepsilon}G_l, & \|G_l\|_2 > C_l.
\end{cases}
\]

### 7.1 这一步在干什么

裁剪的作用是先把每层的“最大信息流量”控住。

如果不裁剪，某一层只要范数特别大，就可能：

- 在后续投影后仍然保留很多强信号；
- 或者导致噪声尺度和梯度尺度严重失衡。

所以裁剪实际上是在给每层先加一个安全阀。

### 7.2 为什么用“相对中位数”而不是固定阈值

代码没有给每层使用完全固定的全局阈值，而是用：

- 当前所有层范数的中位数；
- 再乘以该层自己的 `clip_scale`。

这样做的好处是：

- 不同模型规模、不同 batch、不同任务下能自适应；
- 不至于某些层天生数值大就总被过度裁剪；
- 同时还能保留“敏感层更严、普通层更松”的差异。

## 8. 第四步：低分辨率公共子空间投影

这是 LRB 最核心的一步。

### 8.1 先讲概念

文档叙事里常把这一步说成：

> 把梯度投影到一个公共的、低恢复性的子空间里。

这个说法在思想上是对的，但如果严格对应到当前代码，需要更准确一点：

> 当前实现并不是显式构造一个正交投影矩阵 \(P_l\)，而是实现了一个**低分辨率重建算子** \(T_l(\cdot)\)，用它来保留粗结构、丢掉细节。

这是一个很关键的严谨点。

### 8.2 `pool` 模式

如果模式是 `pool`，那么当前实现做的是：

- 对 1D 张量：先做 `adaptive_avg_pool1d` 把长度压到 \(k \approx n \cdot r_l\)，再线性插值回原长度；
- 对 2D 张量：先做 `adaptive_avg_pool2d` 把矩阵压到 \((k_m, k_n)\)，再双线性插值回原尺寸；
- 对更高维张量：先展平，再按 1D 方式处理。

因此可以把它抽象写成：

\[
G_l^{\text{proj}} = T_{r_l}^{\text{pool}}(\widehat G_l).
\]

这个算子的含义是：

- 保留低频、粗粒度结构；
- 抹掉高频、局部、精细的坐标级变化。

它非常像图像处理中“先缩小，再放大”的效果。

### 8.3 `signed_pool` 模式

当前默认模式不是 `pool`，而是 `signed_pool`。

它的做法是：

1. 先生成一个由随机 \(\pm1\) 组成、但由种子固定下来的符号掩码 \(D_l\)；
2. 对梯度先做符号翻转；
3. 再做 pooling + interpolation；
4. 最后把符号翻转回来。

形式上可以写成：

\[
T_{r_l}^{\text{signed}}(x)
=
D_l \, T_{r_l}^{\text{pool}}(D_l x).
\]

对 2D 情况，代码会分别对行和列使用独立的 Rademacher 符号。

### 8.4 为什么 `signed_pool` 更合理

如果直接做 `pool`，那它强依赖原始坐标系：

- 哪些相邻维度被平均到一起；
- 哪些局部坐标会互相抵消；
- 都是坐标轴固定决定的。

而 `signed_pool` 在 pooling 前先乘随机符号，等价于先做一次轻量的随机混合，再做低分辨率重建。这样更接近：

- “公共随机子空间”
- “固定共享基底”
- “不让原始坐标轴直接暴露私有结构”

所以 `DEFENSE_BASELINES.md` 才会把它描述成更接近“公共随机子空间”的实现。

### 8.5 从 recoverability 角度怎么理解

这一步的作用不是简单压缩通信量，而是主动破坏：

- 样本特异的精细方向；
- 局部尖锐结构；
- 与当前 token/样本强绑定的高分辨率模式。

保留下来的，是一种更模糊、更公共、更粗粒度的梯度版本。

如果用一句最直观的话概括：

> 原始梯度像高清图，投影后的梯度像经过公共模板重建后的低清图。

## 9. 第五步：主要向残差方向加噪

### 9.1 不是“全空间乱加噪”

传统“只加噪”的做法，常常是在整个梯度空间直接加一个噪声张量：

\[
\widetilde G_l = \widehat G_l + Z_l,\quad Z_l \sim \mathcal N(0, \tau^2 I).
\]

这样的问题是：

- 噪声会同时污染有用方向和危险方向；
- 为了防攻击，往往要加很大噪声；
- 一大就容易把训练也打坏。

而 LRB 的代码不是这么做的。

### 9.2 当前实现的噪声构造

先采样一个高斯噪声张量：

\[
Z_l \sim \mathcal N(0, I).
\]

然后用同样的低分辨率算子做一次“可保留部分”：

\[
T_l(Z_l).
\]

再取残差：

\[
R_l = Z_l - T_l(Z_l).
\]

也就是说，只保留噪声里**落在低分辨率公共表示之外**的那一部分。

接着把它重缩放到目标范数：

\[
\xi_l
=
\sigma_l \|\widehat G_l\|_2
\cdot
\frac{R_l}{\|R_l\|_2 + \varepsilon}.
\]

最后输出：

\[
\widetilde G_l = G_l^{\text{proj}} + \xi_l.
\]

### 9.3 为什么说这是“残差方向加噪”

因为代码并不是对整个空间平均加噪，而是：

- 先定义“哪些粗结构是允许保留的”；
- 再让噪声主要落在“没被保留的方向”上。

从设计动机上看，这更符合 recoverability bottleneck 的思路：

- 保留对训练有用的粗结构；
- 重点污染可能泄露细节的剩余空间。

### 9.4 严谨提醒

这里常被口头描述为“正交残差空间加噪”。这个说法在概念上基本成立，但如果严格对应代码，也要加一句限定：

- 当前的 `pool / signed_pool` 更像线性低分辨率重建算子；
- 它不是显式正交投影矩阵；
- 因此这里的“orthogonal residual”更准确地说是**相对于该低分辨率算子的残差部分**。

## 10. 把整个 LRB 写成一个统一公式

把上面几步合起来，当前代码中的第 \(l\) 层 LRB 可以抽象写成：

1. 敏感度：

\[
s_l = (1-w)p_l + w e_l.
\]

2. 参数插值：

\[
r_l = (1-s_l)r_{\text{other}} + s_l r_{\text{sens}},
\]

\[
c_l = (1-s_l)c_{\text{other}} + s_l c_{\text{sens}},
\]

\[
\sigma_l = (1-s_l)\sigma_{\text{other}} + s_l \sigma_{\text{sens}}.
\]

3. 裁剪：

\[
\widehat G_l = \operatorname{clip}(G_l, M c_l).
\]

4. 低分辨率重建：

\[
G_l^{\text{proj}} = T_{r_l}(\widehat G_l).
\]

5. 残差方向噪声：

\[
\xi_l = \sigma_l \|\widehat G_l\|_2
\cdot
\frac{Z_l - T_{r_l}(Z_l)}{\|Z_l - T_{r_l}(Z_l)\|_2 + \varepsilon},
\quad Z_l \sim \mathcal N(0, I).
\]

6. 最终上传更新：

\[
\widetilde G_l = G_l^{\text{proj}} + \xi_l.
\]

最终输出就是

\[
\widetilde G = (\widetilde G_1, \widetilde G_2, \dots, \widetilde G_L).
\]

这个公式很好地对应了代码的真实结构：

- `layer-wise sensitivity`
- `clip`
- `low-resolution public representation`
- `residual-space noise`

## 11. 用伪代码再看一遍

```text
Input: gradients {G_l}, layer_names, args

1. 用层名规则得到每层先验敏感度 p_l
2. 用当前梯度统计得到每层经验敏感度 e_l
3. 混合得到最终敏感度 s_l
4. 计算所有层梯度范数中位数 M

For each layer l:
    a. 根据 s_l 插值得到 keep_ratio r_l
    b. 根据 s_l 插值得到 clip_scale c_l
    c. 根据 s_l 插值得到 noise_scale σ_l
    d. 令 clip threshold 为 C_l = M * c_l
    e. 做梯度裁剪，得到 Ĝ_l
    f. 做低分辨率重建，得到 G_proj_l = T_{r_l}(Ĝ_l)
    g. 采样噪声 Z_l，并保留其残差部分 R_l = Z_l - T_{r_l}(Z_l)
    h. 将 R_l 缩放到 σ_l * ||Ĝ_l||
    i. 输出 G~_l = G_proj_l + scaled(R_l)

Return {G~_l}
```

## 12. 各个参数到底控制什么

下面这张表对应 `args_factory.py` 里的参数。

| 参数 | 默认值 | 作用 | 变大后的直觉效果 |
|---|---:|---|---|
| `defense_lrb_sensitive_n_layers` | `2` | 规定最前面多少层默认视为高敏感 | 更多早层会被重保护 |
| `defense_lrb_keep_ratio_sensitive` | `0.2` | 敏感层保留比例 | 变大表示敏感层保留更多信息，防御更弱、效用可能更好 |
| `defense_lrb_keep_ratio_other` | `0.75` | 非敏感层保留比例 | 变大表示普通层更接近原始梯度 |
| `defense_lrb_clip_scale_sensitive` | `0.5` | 敏感层裁剪阈值缩放 | 变大表示敏感层裁剪更松 |
| `defense_lrb_clip_scale_other` | `1.0` | 非敏感层裁剪阈值缩放 | 变大表示普通层裁剪更松 |
| `defense_lrb_noise_sensitive` | `0.03` | 敏感层残差噪声倍率 | 变大表示敏感层更难恢复，但效用风险也更高 |
| `defense_lrb_noise_other` | `0.005` | 非敏感层残差噪声倍率 | 变大表示普通层也会更强扰动 |
| `defense_lrb_empirical_weight` | `0.6` | 先验与经验校准的混合权重 | 越大越依赖当前 batch 的梯度统计 |
| `defense_lrb_calibration_samples` | `4096` | 校准时每层最多抽多少元素 | 越大校准更稳，但开销更高 |
| `defense_lrb_projection` | `signed_pool` | 低分辨率重建模式 | `signed_pool` 比 `pool` 更接近“公共随机子空间” |

## 13. 为什么它不是普通的 noise / topk / compression

这一点很容易讲混，所以单独说。

### 13.1 它不是纯噪声法

纯噪声法的中心思想是：

- 梯度本身保留；
- 额外加点随机扰动。

而 LRB 的中心思想是：

- 先主动改变梯度的可表达结构；
- 再主要在危险的残差方向上加噪。

所以它不是“以加噪为主”，而是“以 recoverability bottleneck 为主，噪声为辅”。

### 13.2 它也不是单纯通信压缩

`topk` 或 `compression` 的目标通常更偏向：

- 节省带宽；
- 顺便看看是否带来一点隐私收益。

而 LRB 虽然也会压缩信息，但它压缩的重点是：

- 哪些层更敏感；
- 哪些方向更可能泄露；
- 如何把“任务公共结构”和“样本细节结构”尽量分开对待。

所以它的叙事中心是隐私，而不是通信效率。

## 14. 为什么代码和文档里会出现 “LRB-v1 / LRB-v2 / HLRB”

仓库里其实同时存在三个层次的说法：

### 14.1 `LRB-v1`

可以粗略理解为更早的思路：

- 更依赖层名启发式；
- 更偏坐标系上的直接 pooling；
- 研究叙事还没有那么强调动态校准和公共随机子空间。

### 14.2 当前代码里的 `LRB`

从 `DEFENSE_BASELINES.md` 的说法看，当前默认已经是更接近 `LRB-v2` 风格的实现，主要体现在两点：

- 敏感度不再只看结构先验，还融合了当前梯度统计；
- 默认投影模式从 `pool` 升级到 `signed_pool`。

因此可以说：

> 当前代码实现的是一个“v2 风格”的 LRB，而不是纯粹旧版 v1。

### 14.3 `HLRB`

`FL-LLM.md` 里的 `HLRB` 更像研究版、完整版叙事：

- 不只是 backward-side 梯度后处理；
- 还想把 forward-side representation bottleneck 一起做进去；
- 更强调公开校准、公共子空间、跨攻击族 recoverability 最小化。

所以最准确的关系是：

- 当前代码实现：可运行的 LRB 防御版本；
- 文档研究愿景：更完整的 `LRB-v2 / HLRB`。

## 15. 这个方法为什么可能对数据恢复攻击有效

从攻击视角看，很多恢复攻击之所以能工作，是因为原始梯度里同时带有三类东西：

1. 和任务有关的公共信息；
2. 和当前 batch/样本绑定的局部细节；
3. 在某些层中特别清晰的高分辨率结构。

LRB 就是在逐层拆这三件事：

- 用敏感度估计找出最值得重点保护的层；
- 用低分辨率重建保留公共粗结构；
- 用残差方向加噪污染细节部分；
- 用裁剪防止个别层信号过强。

于是攻击者看到的，不再是“高清可逆”的更新，而更像：

- 被模糊过；
- 被公共模板重建过；
- 细节方向又被额外污染过

的版本。

这会同时削弱：

- 对 token/embedding 空间很敏感的攻击；
- 对局部细节结构依赖很强的攻击；
- 对某些特定层极度依赖的 partial-layer 攻击。

## 16. 通俗说明：把它当成“有选择的模糊化”

如果完全不用公式，可以把 LRB 想成下面这个过程。

### 16.1 原始梯度像高清照片

原始梯度里有很多细节：

- 哪几个 token 出现了；
- 哪些位置最重要；
- 某几个样本造成了什么局部变化。

对攻击者来说，这些细节越清楚，就越容易反推原始输入。

### 16.2 LRB 先判断“哪几层最像监控死角”

它不会对所有层一视同仁，而是先问：

- 哪一层最容易泄露样本？
- 哪一层更像在直接记录 token 痕迹？
- 哪一层其实没那么危险？

危险层就重处理，不那么危险的层就轻一点。

### 16.3 然后把危险层从高清变成低清

这一步很像：

- 先把图缩小；
- 再放大回去；
- 看起来尺寸没变，但细节已经没了。

也就是说，梯度还在，形状也还在，但“能不能靠它精确还原原图”这件事已经变难了。

### 16.4 最后专门往“细节槽位”里撒噪声

这一步不是拿噪声到处乱泼，而是更像：

- 主体轮廓尽量保留；
- 专门在纹理、边缘、细节上做破坏。

所以它比全局乱加噪更聪明一点。

### 16.5 一句话通俗总结

`LRB` 干的事就是：

> 先找出最容易泄露隐私的层，再把这些层的梯度变得更模糊、更粗糙，并且专门破坏细节方向，让服务器还能学到大概趋势，但更难把你的原始样本拼回来。

## 17. 严谨地说，它目前还不具备什么

为了避免把方法说得过头，下面这些限制也要讲清楚。

### 17.1 它不是严格 DP 证明的方法

虽然它有裁剪和加噪，但当前实现并没有给出完整的差分隐私预算 \((\varepsilon,\delta)\) 分析，因此不能把它当成标准 DP-SGD 来表述。

### 17.2 它当前不是训练时 hook

仓库当前版本主要是在防御/攻击评估流程里调用 LRB，而不是已经完整插入 `train.py` 训练主循环。

### 17.3 “公共子空间”在当前代码里是近似实现

当前实现里所谓“公共子空间”并不是显式学习出来的正交基，而是：

- `pool`
- 或 `signed_pool`

这类低分辨率重建算子的工程近似。

### 17.4 敏感度校准是轻量的、局部的

当前经验校准依赖的是当前梯度批次的相对统计量，而不是大规模离线 profiling。因此它更像：

- 在线启发式自适应；
- 不是完整、全局、攻击感知的 layer profile 学习。

## 18. 你可以怎么向别人介绍这个方法

如果是写论文式一句话：

> LRB is a layer-wise adaptive gradient defense that suppresses recoverability by combining sensitivity-aware clipping, low-resolution public-subspace reconstruction, and residual-space noise injection.

如果是给组内同学讲一句话：

> 它会先判断哪些层最容易泄露数据，然后把这些层的梯度压成低清版本，并且主要在被丢掉的细节方向上加噪。

如果是更口语一点：

> 不让服务器看到“高清原图”，只让它看到“够训练用的模糊版”。

## 19. 最后总结

当前仓库里的 `LRB` 本质上是一个**分层、自适应、面向 recoverability 的梯度后处理防御**。它的关键不在于“多加了一点噪声”，而在于把防御拆成了三件更有结构的事：

1. 先判断哪层更敏感。
2. 再把敏感层压到低分辨率公共表示里。
3. 最后主要污染那些更可能承载样本细节的残差方向。

因此，`LRB` 的真正思想可以概括为：

> 不是让梯度整体更乱，而是让梯度里“最可恢复的那部分信息”更难恢复。
